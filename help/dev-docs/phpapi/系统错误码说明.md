# 系统错误码说明

## 📋 错误码规范

### 总体规范
- **格式**: 3位数字错误码 + 错误信息
- **分类**: 按照模块和错误类型分类
- **统一响应**: 所有错误都通过统一异常处理返回

### 响应格式
```json
{
    "status": 400,
    "msg": "参数错误",
    "data": null,
    "time": "2024-01-17 15:30:00"
}
```

## 🛠️ 错误处理最佳实践

### 控制器层处理
```php
try {
    // 业务逻辑处理
    $result = $service->doSomething($params);
    return $this->success($result);
} catch (ApiException $e) {
    // 捕获业务异常
    return $this->fail($e->getMessage(), $e->getCode());
} catch (\Exception $e) {
    // 捕获系统异常
    Log::error('系统异常: ' . $e->getMessage());
    return $this->fail('系统繁忙，请稍后重试', 500);
}
```

### 服务层抛出异常
```php
// 业务验证失败时抛出异常
if (!$user) {
    throw new ApiException('用户不存在', 1201);
}

if ($stock < $quantity) {
    throw new ApiException('商品库存不足', 2001);
}
```

### 自定义异常类
```php
class ApiException extends \Exception
{
    public function __construct($message = "", $code = 400, \Throwable $previous = null)
    {
        parent::__construct($message, $code, $previous);
    }
}

class AdminException extends ApiException
{
    // 管理端专用异常
}
```

## 📝 错误码维护规范

### 新增错误码
1. 选择合适的错误码范围
2. 在本文档中记录错误码和说明
3. 在对应的异常处理中添加错误码映射

### 错误码使用原则
1. **唯一性**: 每个错误码对应唯一的错误场景
2. **可读性**: 错误信息清晰明确，便于理解
3. **一致性**: 相同类型的错误使用相同的错误码范围
4. **扩展性**: 为后续功能扩展预留错误码空间

## 🔍 错误排查指南

### 常见错误排查步骤
1. **确认错误码**: 根据返回的错误码定位问题类别
2. **查看日志**: 检查相关业务逻辑的日志记录
3. **验证参数**: 检查请求参数是否符合要求
4. **检查状态**: 确认相关数据的状态是否正常
5. **测试环境**: 在测试环境复现问题并调试
